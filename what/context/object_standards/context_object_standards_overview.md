# Object Standards: Overview

## Three Core Object Types

| Object | Directory | Template | Schema |
|--------|-----------|----------|--------|
| Module | `what/modules/` | `template_module.md` | — |
| Dataset | `what/datasets/` | `template_dataset.md` | `dataset_yaml_schema.json` |
| Lattice | `what/lattices/` | `template_lattice.md` | `lattice_yaml_schema.json` |

**Targets** are a dataset subtype with `dataset_class: target`. They live in `what/datasets/targets/`.

## Module Standard

Modules are self-contained computational units. Key requirements:

### Required Frontmatter

```yaml
type: module
module_type: compute | mcp_server | data_pipeline | utility
status: active | deprecated | planned
inputs:
  - name: input_name
    type: <type_vocabulary_type>
    description: What this input expects
outputs:
  - name: output_name
    type: <type_vocabulary_type>
    description: What this output produces
```

### Quality Criteria

- Clear, single-responsibility purpose
- All inputs and outputs use the canonical type vocabulary (19 types)
- Dependencies explicitly listed
- Version tracked

## Dataset Standard

Datasets describe data collections with storage, lineage, and access metadata.

### Required Frontmatter

```yaml
type: dataset
dataset_class: source | derived | target
status: active | deprecated
storage:
  provider: s3 | minio | gcs | azure | ceph | local | fuse
  bucket: bucket-name
  path: /path/to/data
  format: csv | parquet | pdb | fasta | sdf | json
lineage:
  source: Description of data origin
  transformations: []
```

### `.dataset.yaml` Schema

For deployed datasets, the `.dataset.yaml` file provides machine-readable metadata conforming to `dataset_yaml_schema.json`. Features:

- Multi-cloud storage abstraction (7 providers)
- Lineage tracking with transformation history
- Federation metadata for cross-node data sharing
- FUSE mount specifications

## Lattice Standard

Lattices compose modules and datasets into executable workflows.

### Required Structure

```yaml
type: lattice
nodes:
  - id: node_identifier
    module: module_name
    config: {}
edges:
  - source: source_node
    target: target_node
    data_mapping:
      - from: output_field
        to: input_field
        type: <type_vocabulary_type>
```

### Design Principles

1. **Typed edges** — Every connection specifies data types from the type vocabulary
2. **Composability** — Lattices can reference other lattices as sub-lattices
3. **Reproducibility** — Full configuration captured for replay
4. **Visualization** — Canvas JSON is the visual serialization format

## FAIR Metadata

All deployed objects require FAIR metadata. Two forms:

### Nested (Vault Frontmatter)

```yaml
fair:
  keywords: ["protein", "binder design"]
  license: MIT
  findability:
    identifier: doi:10.xxx
    metadata_catalog: internal
  accessibility:
    access_protocol: https
    authentication: required
  interoperability:
    standard: aDNA v2.1
    format: yaml
  reusability:
    provenance: Generated by pipeline X
    usage_license: MIT
```

### Flat (Transport YAML)

```yaml
fair_keywords: ["protein", "binder design"]
fair_license: MIT
fair_findability_identifier: doi:10.xxx
```

**Minimum requirement**: `keywords` + `license`.

## Type Vocabulary Quick Reference

| Tier | Types |
|------|-------|
| Primitives | `string`, `number`, `boolean`, `path`, `json`, `binary` |
| Structured | `csv`, `dataframe`, `yaml_config`, `parameter_set` |
| Molecular | `protein_sequence`, `pdb_structure`, `sdf_molecule`, `msa_alignment`, `docking_result`, `md_trajectory`, `density_map` |
| Media | `image`, `canvas_json` |

Use these types in module `inputs:`/`outputs:` blocks and lattice `data_mapping` entries.

## Validation

Use `skill_validate_object` to audit objects against their type standard:
1. Frontmatter completeness per type template
2. Type vocabulary compliance in I/O definitions
3. FAIR metadata presence and minimum fields
4. Cross-reference integrity (modules referenced by lattices exist)
